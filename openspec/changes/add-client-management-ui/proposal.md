# 变更：添加前台操作界面（管理控制台）

## 问题背景
当前桌面客户端仅支持查看即将到来的班次，所有管理操作（班次增删改、用户管理、密码重置等）只能通过手动调用 HTTP API 完成，日常维护成本高且容易出错。

## 目标
- 在同一桌面客户端内提供覆盖后端 API 的前台操作界面。
- 保持权限规则与现有 API 一致：任意已认证用户可管理班次，用户管理与审计日志仅管理员可用。
- 操作结果可视化并具备明确错误提示，便于验证与排错。

## 范围内（In scope）
- 班次管理 UI：列表、创建、编辑、删除。
- 用户管理 UI（管理员）：列表、创建、更新姓名/管理员标记、删除、重置密码。
- 人员目录查看 UI（只读）。
- 账号自助：修改密码、退出登录。
- 审计日志查看 UI（管理员）。
- 与现有 API 语义保持一致；如需日志查看，新增只读审计日志查询接口。

## 范围外（Out of scope）
- 新的权限/角色模型或细粒度授权。
- 复杂排班算法、批量导入导出、打印/报表排版。
- 审计日志高级检索、导出、长期归档策略。
- 移动端或独立 Web 管理后台。

## 验收标准（Success criteria）
- 通过 GUI 完成用户与班次的增删改查，结果与 API 返回一致。
- 非管理员无法进入用户管理与审计日志界面。
- 修改密码成功后旧会话失效并要求重新登录。
- 审计日志可在前台以只读列表查看（若纳入）。

## 风险
- UI 功能集中，界面复杂度上升，可能需要拆分组件以保持可维护性。
- 审计日志数据量增长带来性能与隐私风险。
- 权限提示不清晰可能导致误操作或误解。

## 技术选型（默认假设 + 可替代项）
- 默认假设：继续使用现有 Tauri v2 + Vite + Vue + TypeScript + DaisyUI；在同一客户端应用内新增管理控制台界面；如需日志查看，新增只读日志查询 API。
- 可替代项：拆分独立 Web 管理台；或仅保留 API + 脚本操作而不做 GUI；或暂不开放审计日志查看。

## Specs Outline
- **新增** `manage-console-ui`：前台管理控制台的功能与权限要求。
- **修改** `audit-logs`：补充审计日志查询接口（支持前台查看）。
