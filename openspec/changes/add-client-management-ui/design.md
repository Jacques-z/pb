## Context
现有客户端仅提供“查看即将到来的班次”的功能，所有管理操作依赖手动调用后端 API。需求是为现有桌面客户端补齐与 API 对应的前台操作界面，并保持现有权限与鉴权规则不变。

## Goals / Non-Goals
- Goals:
  - 在同一桌面应用内提供班次、用户、人员、审计日志、账号设置的管理入口。
  - 与现有 API 一致的权限行为与错误提示。
  - 以最小 UI 复杂度完成可用管理流程。
- Non-Goals:
  - 引入新的权限/角色体系。
  - 改动排班算法或班次数据模型。
  - 构建移动端或独立 Web 管理后台。

## Decisions
- UI 结构：在现有客户端内新增管理控制台入口，采用分区/分页的方式呈现“班次、人员、用户、审计日志、账号设置”。
- 鉴权与会话：复用现有登录与 token 存储逻辑，统一在请求失败（401）时清理令牌并引导重新登录。
- 数据刷新策略：写操作成功后统一触发列表刷新，避免复杂的本地状态合并。
- 密码处理：在登录、改密、创建用户、重置密码场景中使用客户端 PBKDF2 预哈希（用户名作盐）生成 password_client_hash。
- 审计日志：仅提供只读列表与手动刷新，默认按时间倒序展示。

## Risks / Trade-offs
- 单页功能密集可能导致维护成本上升，必要时拆分组件。
- 审计日志条目增长可能需要后续引入分页或筛选。
- 管理功能集中对权限提示的准确性要求更高。

## Migration Plan
- 变更为增量功能，不涉及数据迁移。
- 若新增审计日志查询 API，需同步文档与权限说明。
